# Plan 03-01: Schema Migration + Shared Utils + Dependency Install

## Objective

Phase 3 상세 페이지 구현의 전제조건 3가지를 처리한다.

1. **스키마 마이그레이션** — `entryDifficulty` → `hiringDifficulty` 필드명 변경 (Prisma 마이그레이션 + 코드 참조 전체 업데이트)
2. **공유 유틸리티 추출** — `parseJsonArray` 헬퍼를 `lib/utils.ts`로 추출하여 여러 컴포넌트에서 재사용 가능하게 함
3. **패키지 설치** — `react-markdown` + `rehype-sanitize` (마크다운 렌더링용)

## Tasks

<task id="T01" type="execute">
  <title>prisma/schema.prisma에서 entryDifficulty → hiringDifficulty 변경 + 마이그레이션 실행</title>
  <read_first>
    - prisma/schema.prisma (Job 모델 전체 — entryDifficulty 필드 위치, 기본값, 타입 확인)
    - .env (DATABASE_URL 값 확인 — file:./dev.db 인지 컨테이너 경로인지)
    - prisma/migrations/ (기존 마이그레이션 이름 패턴 확인)
  </read_first>
  <action>
    prisma/schema.prisma Job 모델에서 `entryDifficulty` 필드를 `hiringDifficulty`로 이름 변경한다.
    필드 타입과 기본값은 그대로 유지한다 (String, @default 값 변경 없음).

    마이그레이션 실행:
      npx prisma migrate dev --name rename_entry_difficulty

    Prisma Client 재생성:
      npx prisma generate

    Docker 환경에서 실행 중인 경우:
      docker compose exec app npx prisma migrate dev --name rename_entry_difficulty
      docker compose exec app npx prisma generate
  </action>
  <acceptance_criteria>
    - prisma/schema.prisma Job 모델에 `hiringDifficulty` 필드가 존재한다
    - prisma/schema.prisma Job 모델에 `entryDifficulty` 필드가 존재하지 않는다
    - prisma/migrations/ 아래에 이름에 "rename_entry_difficulty"를 포함하는 새 디렉토리가 생성된다
    - 해당 migration.sql에 RENAME COLUMN 또는 동등한 구문이 포함된다
    - `npx prisma generate` (또는 migrate dev) 이후 Prisma Client 생성 오류가 없다
  </acceptance_criteria>
</task>

  <read_first>
    - prisma/seed.ts (jobs 배열 전체 — entryDifficulty 키 사용 위치 파악)
    - app/components/JobCard.tsx (Props 타입 선언, 구조분해, className 분기 등 entryDifficulty 모든 사용 위치)
    - app/page.tsx (entryDifficulty 직접 참조 여부 — 있으면 수정 대상)
  </read_first>
  <action>
    1. prisma/seed.ts: jobs 배열 각 항목의 `entryDifficulty:` 키를 `hiringDifficulty:`로 일괄 변경한다. 값("활발"/"보통"/"거의 없음")은 그대로 유지한다.

    2. app/components/JobCard.tsx: Props 타입에서, 구조분해에서, className 분기 로직에서 `entryDifficulty` 참조를 모두 `hiringDifficulty`로 변경한다.

    3. grep으로 남은 참조 확인 후 수정:
       grep -r "entryDifficulty" app/ prisma/ lib/ --include="*.ts" --include="*.tsx"

    4. seed 재실행으로 DB 데이터 갱신:
       npx prisma db seed
       (또는 docker compose exec app npx prisma db seed)

    - `npx prisma db seed` 실행이 오류 없이 완료된다
    - `npx tsc --noEmit` 실행 시 TypeScript 오류가 없다
  </acceptance_criteria>
</task>

<task id="T03" type="execute">
  <title>lib/utils.ts 생성 — parseJsonArray 공유 헬퍼 추출</title>
  <read_first>
    - app/components/JobCard.tsx (parseJsonArray 함수 정의 확인 — 추출할 원본 코드)
    - lib/db.ts (lib/ 디렉토리 존재 확인 및 기존 파일 패턴 참고)
  </read_first>
  <action>
    lib/utils.ts 파일을 신규 생성한다 (lib/ 디렉토리는 lib/db.ts 기준 이미 존재).

    파일 내용:
      export function parseJsonArray(value: string): string[] {
        try {
          const parsed = JSON.parse(value);
          return Array.isArray(parsed) ? parsed : [];
        } catch {
          return [];
        }
      }

    app/components/JobCard.tsx에서 로컬 parseJsonArray 정의를 제거하고 상단에 import를 추가한다:
      import { parseJsonArray } from "@/lib/utils";
  </action>
  <acceptance_criteria>
    - lib/utils.ts 파일이 존재한다
    - lib/utils.ts에 `export function parseJsonArray(value: string): string[]` 시그니처가 존재한다
    - parseJsonArray(유효하지 않은 JSON)이 빈 배열을 반환하는 로직이 존재한다 (try/catch)
    - app/components/JobCard.tsx에 `import { parseJsonArray } from "@/lib/utils"` import가 존재한다
    - app/components/JobCard.tsx에 parseJsonArray 로컬 정의가 존재하지 않는다
    - `npx tsc --noEmit` 실행 시 오류가 없다
  </acceptance_criteria>
</task>
